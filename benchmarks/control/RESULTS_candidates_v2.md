## TL;DR
- **8 candidates** (AdaBelief, MARS, AdEMAMix, Adan, ScheduleFree, ADOPT, Grams, AdamP) built on the shared backend (codec momentum + factored quantized 2nd moment +
  bit-exact foreach + cautious + GC + dtype-safe checkpoint), each with **bit-exact foreach↔per-param
  parity** across {fp32, bf16, int8, 4bit} and a reference-correctness test. ~210 new tests, all green.
- **Two candidates win a rank-1 axis** in the 13-optimizer control battery (C=128, N=2000, 2-seed):
  - **ScheduleFree — 🥇 convergence / time-to-quality** (fastest of all 13 to the quality bar, beating
    fused AdamW), at **2.03 B/param**. *5th overall.* The clear keeper.
  - **ADOPT — 🥇 generalization (lowest train–val gap, +0.0070)** — it **dethroned AdaPNM** on the gap,
    light (2.03 B/p) and the fastest LoRA-step of the candidates. *6th overall.* Caveat: it reaches that
    gap by **underfitting** (highest loss) — which, under our memorization-resistance thesis, is a
    feature worth a real FID test, not a bug.
- The others land mid-pack and are heavier; none beats our in-house **Lion / Adakaon / AdaPNM** on the
  mean. AdaPNM remains the constant-LR champion.
- **The biggest prize is the backend dividend** (below): 7 independent implementations converged on the
  same handful of consolidations and one genuinely new reusable primitive (a schedule-free mixin).

## ⚠️ The lesson (loss↔gap trap, re-confirmed at scale)
I first tuned every candidate by **held-out loss on the short C96/N800 proxy**. For AdaBelief that drove
β2→0.95 (lowest loss) — but at full scale that config was the **gap-worst of all 13** (+0.0240). Re-tuning
for the gap (β2=0.999, `AdaBelief-b999`) cut the gap **−35 % to +0.0155** with loss barely moving
(0.0742→0.0780). **The proxy-loss optimum is the gap-pessimum.** This is the profiler's documented
degenerate-pick caveat, now confirmed on the real battery: for the small-data thesis, tune candidates on
the **gap / Pareto**, not loss — and the short proxy's gap is too insensitive to tune on (the spread only
opens at C128/N2000). Final judge remains a **real LoRA + FID/KID** (the proxy ranks objective
overfitting, not perceptual fidelity).

## 🔧 Backend dividend (the real prize — proposals for review, not yet applied)
Seven independent implementations converged on the same findings:
1. **Consolidate the codec read/store helpers.** `_dequant_one/_store_one/_dequant_stacked/_store_stacked`
   are copy-pasted across `adapnm`, `adabelief`, `mars`, `ademamix`, `adan` (and Schedule-Free's z-store).
   Lift a prefix-parameterized version into `_momentum_codec.py` (~120 lines saved per optimizer).
2. **Kill the fp32-aliasing footgun class-wide.** Reading an fp32 momentum buffer via `.float()` returns
   the *same* tensor; an in-place update then corrupts stored state. **5 of 7 candidates hit this.** The
   fix every agent found: route momentum through the codec `ema_one`/`ema_stacked` (they `.clone()` fp32) —
   exactly what `adopt.py`/`adakaon.py` do correctly. Make that the only sanctioned path + add a
   `_backend._fresh_fp32()` helper.
3. **A schedule-free / iterate-averaging MIXIN (genuinely new, highest-value).** Schedule-Free's z/x/y +
   `train()`/`eval()` averaging is optimizer-agnostic (the official ships it as a wrapper around *any* base
   optimizer). A kaon mixin (train/eval swap + ckp1 averaging + z-codec storage) would give **AdaPNM,
   Adakaon, AdaMuon schedule-free variants for free** — and SF already won convergence here.
4. **`blend_batched_(pviews, target, weight, bf16_method)`** so the SF train/eval lerp swap is
   stochastic-rounding/kahan-aware (currently a plain `lerp_` — a minor bf16 fidelity gap).
5. **A shared `_schedulers.py`** for hyperparameter warmups (AdEMAMix's α/β3 half-life warmup; reusable for
   Prodigy D-warmup, AdaMuon ns-warmup).
6. **Batched multi-buffer dequant** for heavy optimizers (Adan's 3 quantized buffers = 3× dequant→requant
   launches/step the backend doesn't amortize).
7. **Document the codec invariant**: `ema_one` expects the update in the param's *original* (not matrixized
   `[R,C]`) shape — an undocumented trap that bit two agents.

Also shipped this campaign: `battery.py` now calls `opt.train()/opt.eval()` around the step/evals when an
optimizer exposes them (guarded — every torch optimizer's run stays bit-identical), so Schedule-Free-style
optimizers are measurable in the battery.

## Findings worth keeping regardless of which candidates ship
- **AdEMAMix's slow EMA m2 is precision-sensitive** (int8 +0.9 %, 4bit +2.4 % held-out vs bf16) — long-horizon
  EMAs want bf16. Suggests a future per-buffer dtype knob (cheap m1 + bf16 m2).
- **Adan int8 momentum is lossless** (te 0.0876 fp ≈ 0.0878 int8) — halves its 3-buffer footprint for free.
- **kozistr ScheduleFreeWrapper has a real bug** (`step**lr` should be `step**r`) — not carried; we used the
  official AdamWScheduleFree closed form.

## A note the proxy keeps teaching us (Grams + ADOPT)
The two sign/normalize-based candidates (Grams, ADOPT) **flip behaviour between the short proxy and the
full-scale battery**: on C96/N800 both looked like low-gap regularizers; at C128/N2000 ADOPT *stayed*
low-gap (it genuinely underfits) while **Grams reversed** into a loss-aggressive memorizer (gap +0.0210).
Lesson reinforced: short-proxy gap/loss does not extrapolate for these mechanisms — only the full-scale
battery (and ultimately FID) decides. Don't trust a candidate's proxy gap.
